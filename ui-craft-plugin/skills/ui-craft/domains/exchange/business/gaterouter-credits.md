## 区块 0: 协作指南

> 基于 EvoSkill 分析的工作流程优化，避免重复踩坑

### 0.1 信息源优先级

| 优先级 | 信息源 | 用途 | 说明 |
|--------|--------|------|------|
| 1 | Figma 设计稿 | 视觉真值 | 可用 MCP 获取精确 Token |
| 2 | 高清截图 | 视觉参考 | 至少 1920px 宽 |
| 3 | 现有 HTML | 实现参考 | 用于校准和增量修改 |
| 4 | PRD 文档 | 功能逻辑 | 不作为视觉依据 |
| 5 | 网站 URL | 仅供参考 | WebFetch 无法获取视觉 |

**规则**：当多个源冲突时，以高优先级源为准。用户应在开始时声明「以 X 为准」。

### 0.2 用户准备清单

开始生成页面前，用户需要提供：

```markdown
## 必须提供（至少一项）
- [ ] Figma 链接（最佳）
- [ ] 高清截图（次选）

## 必须确认
- [ ] 主题：亮色 / 深色
- [ ] 品牌色 HEX 值
- [ ] 单一真值声明：「以 Figma 为准」或「以截图为准」

## 可选提供
- [ ] PRD 文档（功能逻辑）
- [ ] 现有 HTML 参考
```

### 0.3 AI 校准检查点

| 阶段 | 检查项 | 通过条件 |
|------|--------|----------|
| 开始前 | 视觉参考 | 有 Figma 链接或截图 |
| 开始前 | 主题确认 | 用户明确说「亮色」或「深色」 |
| 生成后 | 主题一致 | HTML 主题与规范一致 |
| 生成后 | 品牌色一致 | `--brand-primary` 与设计稿一致 |
| 迭代后 | 规范同步 | HTML 稳定后更新本文档 |

**规则**：任一检查项未通过，AI 应主动询问用户，而非猜测。

### 0.4 增量迭代流程

```
Phase 1: 首次生成
  /ui-craft --calibrate <figma/截图>  ← 先校准
  /ui-craft --domain exchange         ← 再生成

Phase 2: 迭代修改（优先增量）
  /ui-craft --patch <html>            ← 增量补丁（推荐）
  /ui-craft --css-patch <html>        ← CSS 快速补丁
  ❌ 避免全量重写

Phase 3: 规范同步
  HTML 稳定后 → 更新本文档的 Token/组件/变更日志
```

### 0.5 历史教训（反馈沉淀）

| 日期 | 失败类型 | 根因 | 改进措施 |
|------|----------|------|----------|
| 2026-03-14 | 深色主题错误 | WebFetch 无法获取视觉，AI 猜测 | 强制要求截图或 Figma |
| 2026-03-14~18 | 5 次全量重写 | 每次发现问题都重写 | 优先使用 `--patch` |
| 2026-03-18 | 规范与 HTML 脱节 | 改 HTML 没更新规范 | 每次迭代后同步规范 |

### 页面导航规则

> **可逆性原则**：所有页面跳转必须支持双向导航，用户可以原路返回

**强制规则：**

1. **双向链接**：页面 A 链接到页面 B 时，页面 B 必须有链接返回页面 A
2. **相对路径**：同目录下的页面使用相对路径（如 `gaterouter-credits-v4.html`），确保本地预览可用
3. **Active 状态**：当前页面的导航项必须显示 `active` 状态
4. **一致性**：所有页面的侧边栏导航结构必须完全一致

**导航映射表：**

| 页面文件 | 导航项 | Active 状态 | 可跳转到 |
|----------|--------|-------------|----------|
| `gaterouter-credits-v4.html` | 额度 | ✓ | 额度-1 |
| `gaterouter-credits-1.html` | 额度-1 | ✓ | 额度 |

### 交互规则

1. 点击 LoginModal「连接钱包」→ 登录弹窗向左滑出，钱包弹窗从右侧滑入
2. 点击返回按钮 → 钱包弹窗向右滑出，登录弹窗从左侧滑入
3. 点击关闭按钮 → 关闭弹窗，返回页面
4. 点击 Tab 切换 → 切换钱包内容面板
5. 点击二维码按钮 → 展开/收起二维码展示区，同时隐藏/显示底部下载链接
6. 点击钱包选项 → 触发连接流程
