# 反馈沉淀协议（Feedback Loop）

> 建立从"人类 Review 发现问题"到"规则更新"的闭环，确保约束体系持续进化。

---

## 三级沉淀路径

| 级别 | 名称 | 载体 | 触发条件 |
|------|------|------|----------|
| Level 1 | 口头建议 | Review 评论 / 对话记录 | 首次出现 |
| Level 2 | 文档约束 | 更新 `rules.md` 的 check/fix 字段 | 同类问题出现 2 次 |
| Level 3 | 扫描约束 | 新增到 `verification-protocol.md` 的 Pass 1 扫描项 | 同类问题出现 3 次 |

---

## 沉淀流程

### Level 1: 记录

```
发现问题 -> 记录在 Review 评论或对话中
           -> 标注"潜在规则"标签
           -> 不修改任何约束文件
```

### Level 2: 文档化

```
同类问题第 2 次出现
  -> 确认问题模式（是通用的，不是个案）
  -> 更新 rules.md 中对应规则的 check/fix 字段
  -> 如果是新规则类型，在 rules.md 末尾新增规则条目
  -> 同步更新自检清单
  -> 记录变更到 SKILL.md 的"规则变更日志"
```

### Level 3: 自动化

```
同类问题第 3 次出现
  -> 在 verification-protocol.md 的 Pass 1 扫描表新增检查项
  -> 定义检查方法和 severity
  -> 确保后续输出自动被扫描发现
  -> 记录变更到 SKILL.md 的"规则变更日志"
```

---

## 问题分类

| 类别 | 示例 | 典型沉淀路径 |
|------|------|-------------|
| 视觉一致性 | 同页面按钮圆角不统一 | 已有 R11，补充 check 细节 |
| 语义错误 | accent 用于非金额场景 | 已有 R4，强化 check |
| 性能问题 | 过多 box-shadow 导致重绘 | 新增 R28+ |
| 品牌合规 | 使用了非品牌色 | 已有 R3，补充 fix |
| 可访问性 | 缺少 aria-label | 扩展 R27 |
| 新发现模式 | 某类布局反复出错 | 新增规则 |

---

## 变更记录要求

每次 Level 2 或 Level 3 变更，必须在 SKILL.md 的"规则变更日志"中记录：

| 字段 | 说明 |
|------|------|
| 日期 | 变更日期（YYYY-MM-DD） |
| 规则 | 影响的规则编号（如 R2、新增 R25） |
| 变更 | 具体变更内容（新增 check/更新 fix/新增扫描项） |
| 来源 | 触发来源（Review 评论/用户反馈/自检发现） |
